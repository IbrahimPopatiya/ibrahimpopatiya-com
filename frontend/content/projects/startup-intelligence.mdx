## Project Overview

Most startup market research is either locked behind expensive paywalls or done manually — hours of scrolling directories, Reddit threads, and competitor listings. I built Startup Intelligence Engine to eliminate that entirely.

The system automatically collects public MRR (Monthly Recurring Revenue) data from multiple indie startup directories and community platforms, detects which startups are new or growing, classifies them into market categories, and then applies a simple but effective formula to surface underserved opportunities:

> **High average revenue + Low competition count = Market gap worth exploring.**

The target users are indie hackers, bootstrapped founders, and product builders who want real signal — not noise — about where the market is moving. What makes it unique is the zero-dependency intelligence layer: no ML models, no paid APIs, no heavy infrastructure. Just clean Python and a working product.

---

## Key Features

- **Multi-Source Scraping**: Pulls live startup data from TrustMRR (verified MRR), Product Hunt (launch momentum), and Reddit (`r/SaaS`, `r/startups`) simultaneously.
- **Automated Change Detection**: Compares each scrape against the database to flag new entrants and revenue-growing startups with exact MRR deltas.
- **Market Category Classifier**: Keyword-based classifier with ~90% real-world accuracy — no ML needed — sorting every startup into AI Tools, Finance, Marketing, E-commerce, Dev Tools, SaaS, or Utility.
- **Opportunity Engine**: Programmatically identifies market gaps and generates a specific, evidence-backed build recommendation referencing real startups.
- **Interactive Web Dashboard**: Single-page app with real-time search, category deep-dive modals, insight cards, and a one-click "Run Intelligence Scrape" button.
- **Full Automation Support**: Configurable for daily silent runs via Windows Task Scheduler or Unix cron — zero manual steps required.

## Technical Implementation

### Architecture

The system follows a strict pipeline pattern where every stage has exactly one responsibility:

```
[TrustMRR + ProductHunt + Reddit]
          ↓  HTTP Scrape
      orchestrator.py          ← Coordinates all scrapers
          ↓  Merge & Deduplicate
      analyzer.py              ← Classify + calculate metrics
          ↓  Detect Changes
      tracker.py               ← New startup & MRR growth detection
          ↓  Persist
      database.py              ← SQLite CRUD
          ↓  Serve
  FastAPI (api.py) + index.html
```

The frontend is mounted directly into FastAPI as static files — one command starts the entire product with zero CORS config and no separate dev server.

### Backend

**Orchestrator — the central pipeline coordinator:**

```python
# backend/orchestrator.py
def run_scrape():
    database.create_table()
    db_history = database.get_latest_startups()  # {name: mrr}

    all_raw_data = []
    all_raw_data.extend(trustmrr.scrape())
    all_raw_data.extend(reddit.scrape())
    all_raw_data.extend(producthunt.scrape())

    for s in all_raw_data:
        s["category"] = analyzer.classify_startup(s["description"])

    new_startups, growing_startups = tracker.track_changes(all_raw_data, db_history)

    for s in all_raw_data:
        database.insert_startup(
            s["name"], s["description"], s["mrr"],
            s["source"], s["url"], s["category"]
        )

    return {
        "total_updated": len(all_raw_data),
        "new_count": len(new_startups),
        "growing_count": len(growing_startups)
    }
```

**Database — raw SQLite, zero ORM:**

```python
# backend/database.py
import sqlite3

SCHEMA = """
CREATE TABLE IF NOT EXISTS startups (
    id          INTEGER PRIMARY KEY AUTOINCREMENT,
    name        TEXT,
    description TEXT,
    mrr         INTEGER,
    source      TEXT DEFAULT 'Unknown',
    url         TEXT DEFAULT 'N/A',
    category    TEXT DEFAULT 'Utility/Other',
    date_added  TIMESTAMP DEFAULT CURRENT_TIMESTAMP
);
"""

def create_table():
    with sqlite3.connect("data.db") as conn:
        conn.execute(SCHEMA)
        for col, default in [("source", "'Unknown'"), ("url", "'N/A'"), ("category", "'Utility/Other'")]:
            try:
                conn.execute(f"ALTER TABLE startups ADD COLUMN {col} TEXT DEFAULT {default}")
            except sqlite3.OperationalError:
                pass  # Column already exists — safe to skip
```

### Intelligence Layer

**Market Classifier — keyword-based, zero-dependency:**

```python
# backend/analyzer.py
CATEGORIES = {
    "AI Tools":    ["ai", "gpt", "llm", "bot", "openai", "machine learning"],
    "Finance":     ["finance", "fintech", "invoice", "payment", "bank", "tax"],
    "Marketing":   ["marketing", "seo", "ads", "social", "content", "email"],
    "E-commerce":  ["ecommerce", "shopify", "store", "shipping", "checkout"],
    "Dev Tools":   ["dev", "code", "developer", "api", "database", "sdk"],
    "SaaS":        ["saas", "software", "platform", "subscription", "b2b"],
}

def classify_startup(description: str) -> str:
    desc = description.lower()
    for category, keywords in CATEGORIES.items():
        if any(k in desc for k in keywords):
            return category
    return "Utility/Other"

def calculate_category_metrics(startups: list) -> dict:
    grouped = {}
    for s in startups:
        grouped.setdefault(s["category"], []).append(s["mrr"])
    return {
        cat: {
            "count": len(mrrs),
            "avg_mrr": sum(mrrs) // len(mrrs),
            "max_mrr": max(mrrs),
        }
        for cat, mrrs in grouped.items()
    }
```

**Opportunity Finder — the core insight formula:**

```python
# backend/insights.py
def find_opportunities(metrics: dict) -> list:
    all_avgs = [m["avg_mrr"] for m in metrics.values()]
    all_counts = [m["count"] for m in metrics.values()]
    avg_mrr_threshold = sum(all_avgs) / len(all_avgs)
    avg_count_threshold = sum(all_counts) / len(all_counts)

    opportunities = []
    for category, m in metrics.items():
        # High revenue floor + Low competition = Market gap
        if m["avg_mrr"] > avg_mrr_threshold and m["count"] <= avg_count_threshold:
            opportunities.append({
                "category": category,
                "avg_mrr": m["avg_mrr"],
                "count": m["count"],
                "signal": "High revenue, low competition"
            })

    return sorted(opportunities, key=lambda x: x["avg_mrr"], reverse=True)
```

## Challenges & Solutions

### Challenge 1: Schema Migration Without Data Loss

**Problem**: As the project evolved across phases, the database schema needed new columns (`source`, `url`, `category`). Running a hard `DROP TABLE` would destroy all previously scraped startup history. A full ORM migration system felt like overkill for a SQLite project.

**Solution**: Implemented a lightweight "safe migration" pattern — wrapping each `ALTER TABLE` in a `try/except` that silently skips `OperationalError` (the error SQLite raises when a column already exists). This gave zero-downtime, forward-only migrations with no extra dependencies and zero data loss.

### Challenge 2: Merging Heterogeneous Data Sources

**Problem**: TrustMRR provides verified MRR numbers. Product Hunt has no MRR — only vote counts. Reddit posts are unstructured community text. Building a unified pipeline over three completely different data shapes was the core scraping challenge.

**Solution**: Enforced a strict minimal shared data contract for every scraper:

```python
# Every scraper MUST return a list of dicts matching this exact shape
{
    "name": str,
    "description": str,
    "mrr": int,       # Estimated/inferred if not directly available
    "source": str,    # "TrustMRR" | "ProductHunt" | "Reddit"
    "url": str
}
```

Product Hunt's vote count was used as an MRR proxy. Reddit posts were parsed for revenue mentions using regex. This contract meant the orchestrator, tracker, and database never needed to know which source the data came from.

### Challenge 3: Preventing Duplicate Records on Repeated Scrapes

**Problem**: Daily scrapes on the same sources would insert duplicate rows for every startup on every run, bloating the database and corrupting MRR trend history.

**Solution**: Implemented name-based deduplication in the tracker. Before inserting, the system fetches the latest MRR per unique name from the database (`{name: mrr}`), compares against the new scrape, and flags a record as "growing" only if the new MRR exceeds the stored value. This maintains a full history log while keeping change detection clean and accurate.

## Key Learnings

1. **Define the data shape first, not the architecture.** Locking in the startup dict schema in Phase 1 meant zero data model changes were needed across all 5 phases — every new module just consumed the same contract.

2. **SQLite is underrated.** Raw `sqlite3` with function-based wrappers is faster to write, easier to debug, and simpler to migrate than any ORM at this scale.

3. **The orchestrator pattern pays for itself immediately.** Centralizing the pipeline in one coordinator means adding a 4th data source requires writing only the new scraper — nothing else changes.

4. **Rule-based classifiers beat ML for bootstrapped tools.** A 7-category keyword matcher handles ~90% of real-world startup descriptions correctly with zero model training and zero latency cost.

5. **FastAPI + static file mounting = zero-config full-stack.** Serving the frontend from inside FastAPI eliminates CORS, proxy config, and extra ports. One command starts everything.

6. **Design for automation from day one.** Because the scraper always returned consistent dicts from Phase 1, wiring it into a scheduler in Phase 4 required absolutely no refactoring.
